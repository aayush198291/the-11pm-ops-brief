01The problem

Every supply-chain operator wakes up to the same dread question: "what hit overnight?" The answer lives in two unconnected worlds — internal data (in-flight shipments, at-risk revenue, active disruptions) and external signals (weather, port congestion, news pulse, carrier diversions, fuel prices, border waits, earthquakes, hurricanes, public-health alerts). Nobody reads all of it at 11 PM. Most of it never reaches the operator at all.

The 11 PM Ops Brief is an autonomous multi-agent system that fuses all of it into a single grounded, executive-grade page — with proposed write-back actions — every night.

03The 7 nodes of the supervisor graph

Each subagent is its own create_react_agent with a curated tool subset. The Supervisor's Plan is a Pydantic-validated structured output — the graph walks it deterministically. Routing is plan-driven, not LLM-driven at each hop, which keeps traces clean and reproducible.

🧠

Supervisor

planner · 1 LLM call

Reads the user query, emits an ordered plan: ["data_analyst","signal_scout","historian","composer","critic"] for a full brief; ["data_analyst"] for a count question.

tools: none — pure planner · output: Plan (Pydantic)
📊

DataAnalyst

internal data via Genie MCP

Queries the Genie space (UC Metric Views). Drives the query_space_* / poll_response_* polling loop. Returns numbers + the underlying Genie SQL.

tools: Genie MCP tools + get_current_time
📡

SignalScout

17 external feeds · parallel

Parallelized: asyncio.gather fans out across all 17 signal tools, then a single LLM call compresses results into 3-6 bullets. ~26s (was ~90s).

tools: 17 fetch_* + 3 UC AI Functions
📚

Historian

memory + Vector Search recall

Recalls past briefs via Vector Search (briefs_archive_index) — semantic match returns recurring themes. Memory tools persist user prefs across sessions.

tools: recall_similar_briefs · memory · Genie
✍️

Composer

brief + write-back proposals

Synthesizes data + signals + history into executive markdown. Calls action tools to populate the UI Action Queue. Emits a JSON sidecar with severity score and pending actions.

tools: compose_brief · 4 action tools · UC AI Fns
🔍

Critic

quality gate · LLM-only

QCs the draft for groundedness, action specificity, severity calibration. Emits PASS/FAIL — Publisher appends the verdict to the final brief.

tools: none — pure judge
📤

Publisher

canonical final emit

Picks the right final message from accumulated state (composer's brief > critique > data findings). Ensures the chat UI's "last message" is always the answer the user wants.

tools: none — pure router
🎼

Plan-driven routing

graph edges

The Supervisor picks the plan once. route() steps through it deterministically. Once exhausted, Publisher runs once → END. No expensive supervisor re-evaluation per hop.

file: agent_server/supervisor.py:600

05Unity Catalog AI Functions Stage 2 — AI Functions

Three governed SQL UDFs in dnb_hackathon_west_2.ops_brief, each powered by ai_query or ai_classify. Registered via setup_uc_functions.py, granted EXECUTE, and bound as agent tools via UCFunctionToolkit. Lineage, ACL, and billing all flow through Unity Catalog.

🏷️

classify_disruption_severity

ai_classify → critical | elevated | nominal

Buckets free-text disruption descriptions. Used by Composer + SignalScout for consistent severity tagging across heterogeneous signal sources.

SELECT classify_disruption_severity(
  'Severe winter storm at MEM hub —
   200mph winds, multiple cancellations.'
) → 'critical'
📝

summarize_signal_pulse

ai_query · executive sentence

Compresses N raw signal blurbs into one executive sentence with the most material concern. Used by SignalScout to keep the brief tight.

→ "Seismic activity across multiple
regions overnight; M5.2 quake 49km
ESE of Adak, Alaska (Aleutians)
accompanied by lesser M4.7 + M4.3…"
⚖️

carrier_risk_score

ai_query · returns JSON

0-100 risk score for shipments tonight given a carrier's recent disruption history. JSON output ({score, reasoning}) so downstream code can parse.

  "score": 78,
  "reasoning": "Active severe T-storm
  warning over TN + two MEM hub

06Action layer — write-back with HITL approval

The difference between a report and an operational system. The Composer proposes 2-4 actions per brief; each lands in the UI's Action Queue with Approve / Reject buttons. On approve, the action commits (in production: Jira, Slack, carrier comms; here: Delta + JSONL audit trail).

🚦

propose_action

generic write-back

Any imperative-verb-led action with severity, owner_role, impact estimate, related_objects. The canonical action proposal — Composer calls this 2-4× per brief.

🎫

file_disruption_ticket

Jira-shape draft

Drafts a disruption-tracking ticket with affected carriers, FCs, lanes, and severity. Returns the draft (not yet sent — awaiting approval).

📧

draft_carrier_email

outbound comms draft

Drafts an email to a named carrier (FedEx/UPS/etc.) with subject, body markdown, impacted-shipment count. Lands in the Action Queue.

📒

log_decision

decisions_log audit trail

Appends a brief-level decision (headline, severity, themes, rationale) to the decisions log — both an in-memory ring buffer and a JSONL file.

Action persistence

Every proposed action is double-written: (a) into an in-process deque(maxlen=50) the UI reads via GET /actions/queue, and (b) into /tmp/ops_brief/actions.jsonl for offline audit. Approve / Reject events append a new row with the operator's identity and timestamp.

07Vector Search recall (Historian) Stage 4 — Memory + MCP

14 nights of synthetic past briefs in a Delta table, embedded with databricks-gte-large-en, indexed in Vector Search. The Historian semantically queries past briefs and surfaces recurring themes.

  • VS index briefs_archive_index · Delta-Sync · TRIGGERED · embedding on body_markdown

08Operational Console UI

Not a chatbot. A 4-pane operational console — Palantir Foundry style, dense, light-theme-only, Nunito Sans + ShipBob design tokens. Single self-contained HTML page (~823 lines), no external assets except Google Fonts and marked.js for markdown rendering.

The 4 panes

  • Left rail — Signals. 17 live tiles (status dot · count · last-updated) refreshed every 60s via /signals/latest
  • Center — Brief. Rendered markdown with clickable object chips [OBJ:FC-MEM] and provenance chips [CITE:query-id]
  • Right rail — Actions. Approve/Reject buttons. Card flips to ✓ APPROVED on commit.
  • Bottom — Timeline. Live supervisor → subagent → tool call transitions, color-coded by actor

Investigation drill-down

Click any object chip in the brief — [OBJ:FC-MEM], [OBJ:CARRIER-FEDEX], etc. — a slide-in panel pulls live properties from the SQL warehouse:

  • Properties: capacity, in-flight, at-risk, region
  • Links: top carriers, active disruptions, related shipments
  • Chained navigation: click any linked chip → new card
  • Foundry-style Object Cards over Databricks Apps

Severity gauge + JSON sidecar

The Composer emits a trailing ```json {severity, pending_actions}``` block. The UI parses it, strips it from the displayed brief, animates the severity gauge from 0→7.2, and populates the Action Queue. Green <4, amber 4-7, red ≥7.
